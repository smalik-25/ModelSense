# ModelSense eval harness

Measures the ModelSense agent on a 50-task golden set with deterministic
trajectory scoring, an LLM-as-judge for context fidelity, and a CI regression
gate. Python 3.12, managed with uv.

## What it measures

Per run (PROJECT_PLAN section 2): task completion rate, tool-selection accuracy,
argument validity, mean/p95 latency, mean cost per task, context fidelity, and
guardrail compliance.

## Pipeline

```
golden/tasks/*.yaml   50 tasks: prompt, model, expected tools + arg matchers,
                      assertions, budgets, category, difficulty
        |
        v  (evals run, LIVE)
runner  --> POST /chat SSE (the real deployed path) + /chat/approve for gated tools
        |
        v
trajectory JSON  --> deterministic scorers  +  Haiku judge (context fidelity)
        |
        v
Parquet results  +  markdown report + charts (reports/)
```

The **CI gate** (`evals gate`) replays committed trajectories in `fixtures/`
through the deterministic scorers only. It never calls a live API.

## Golden answers are computed, not typed

`golden/reference.json` is generated by the TypeScript server (`make golden`)
from the committed GLBs using the exact domain logic the MCP server runs. Tasks
reference values by dotted path (e.g. `DamagedHelmet.totals.triangles`), so a
model change plus a reference regen keeps every assertion honest. `evals
verify-golden` fails if any ref stops resolving.

## Commands

```bash
make sync            # create the venv, install deps
make golden          # regenerate reference.json from the GLBs (runs the TS server)
make verify          # validate the golden set against reference.json
make gate            # CI regression gate over recorded fixtures (offline)
make test            # pytest (scorers, loader, gate, report)
make lint            # ruff

# LIVE run against a running agent (spends tokens):
make run ARGS="--agent-url http://localhost:8787 --yes"
make run ARGS="--category lookup --limit 3 --yes"     # a cheap smoke subset
```

`evals run` prints a cost estimate and refuses to spend without `--yes`.

## Fixtures

`fixtures/trajectories/*.json` are the **recorded live baseline** (50 real agent
trajectories from the run in `reports/`), replayed by the CI gate. Two truck-count
fixtures (`lookup-truck-triangles`, `lookup-truck-drawcalls`) were updated in place
to the corrected numbers after the mesh-instancing stats fix; a fresh live run
refreshes all 50 as new recordings. Regenerate them from a fresh run:

```bash
make run ARGS="--agent-url <url> --save-fixtures --yes"
```

`scripts/seed_fixtures.py` can regenerate synthetic seed trajectories from
`reference.json` as a fallback when no live run is available (e.g. bootstrapping the
gate before the first baseline). `baseline.json` holds the regression floors the
gate enforces.

## Scoring model

- **Tool selection**: longest-common-subsequence of expected vs actual tool
  names, order-preserving; extra tools are allowed.
- **Argument validity**: per-arg matchers (`ref`, `regex`, `contains`, `one_of`,
  `approx`) against the tool inputs.
- **Assertions**: outcome checks on the final answer, emitted scene commands, and
  tool outputs (numbers resolved from `reference.json`, measurements computed from
  bbox facts).
- **Guardrail invariant**: a gated tool must never succeed without an approval.
- **Completion**: assertions pass AND (for non-guardrail) all expected tools
  present in order AND within the step budget; for guardrail, assertions pass AND
  the safety invariant holds.
- **Context fidelity**: Haiku rates whether the answer is grounded in the tool
  outputs. Live runs only; the CI gate runs deterministic scorers with the judge
  off, so it never calls the API. A judged run can persist its per-task score into
  the trajectory (`context_fidelity`), which `evals score` reuses to reproduce the
  number offline. The 4.48/5 headline comes from the live baseline run, not the CI
  gate.
